# Game spec: Wordle clone with hints

## 1. Overview

A single-page, browser-based word-guessing game, modeled on Wordle's core loop and visual language. No backend, no login. All state lives client-side.

## 2. Look and feel

Match the original Wordle conventions:
- Grid of letter tiles, one row per guess.
- On submit, tiles flip/reveal to show correctness.
- Colors: green = correct letter, correct position. Yellow = correct letter, wrong position. Gray = letter not in word.
- On-screen QWERTY keyboard below the grid, mirroring the same three-color feedback per letter.
- Physical keyboard input also works (type letters, Enter to submit, Backspace to delete).

## 3. Difficulty selection

On entry, before the first guess, the player chooses a word length: 5, 6, or 7 letters. This determines:
- Grid column count (equal to word length).
- Which word pool is used (see Section 4).
- Number of guess rows: 6, regardless of word length (standard Wordle allowance).

## 4. Word data source

Word lists are NOT hardcoded or generated at runtime. They are loaded from static JSON files, one per supported length, following this naming convention:

- `words-5.json`
- `words-6.json`
- `words-7.json`

These files will be supplied separately (not generated by the build agent). Each file contains the top 2,000 most commonly used words of that letter length. Expected schema (to confirm once files are provided, but build against this shape):

```json
[
  { "word": "APPLE", "definition": "A round fruit with red, green, or yellow skin." },
  { "word": "BRAVE", "definition": "Showing courage in the face of danger or pain." }
]
```

Requirements on the game logic side:
- The daily/session answer is chosen at random from the relevant file's word list.
- Valid guesses are restricted to words present in that same file (no separate "acceptable guess" list; if it's not in the top-2000 file, reject it as "not in word list").
- If a JSON file is missing or fails to load, show a clear error state rather than falling back to a hardcoded list.

## 5. Hints (two types)

**Hint type 1: Letter reveal (keyboard only)**
- Button reveals one letter that is in the answer AND that the player has not yet guessed/used.
- The reveal only highlights that letter yellow on the on-screen keyboard.
- It does NOT reveal or confirm the letter's position in the word, and does NOT place it into the grid.
- Once a letter has been surfaced this way, it should not be picked again by this hint for the same game.

**Hint type 2: Word meaning**
- Button reveals the dictionary definition of the current answer, sourced from the `definition` field in the JSON word file.
- Purely informational; does not affect grid or keyboard state.

Both hint types are available independently (using one does not consume or disable the other). Exact usage limits (e.g. one use per game vs. unlimited) are open for Roy to decide — flag this as a confirm-before-build item.

## 6. Input feedback

- Valid submission (correct length, in word list): proceeds normally with the flip/reveal animation.
- Invalid submission (wrong length, or not in the word list for the current difficulty): the on-screen keyboard gives a brief "wriggle" (shake) animation as feedback. No modal, no blocking dialog.

## 7. Persistence (resume without login)

- Game state persists in the browser via `localStorage` (or similar client-side storage) — no account or server required.
- On revisit, the player resumes exactly where they left off: same answer, same difficulty, same guesses made so far, same keyboard state, same hint usage.
- Persistence should be scoped per difficulty (switching difficulty starts/resumes that difficulty's own saved state, not a shared one) — flag this as a confirm-before-build assumption.
- A finished game (win or loss) should remain visible in its completed state on return, not silently reset, until the player starts a new game.

## 8. Explicit non-goals (for this version)

- No user accounts, no server, no leaderboard, no analytics.
- No daily-puzzle-for-everyone mechanic (each player's session is independent) unless later requested.
- No word list generation/sourcing logic in the game itself — that is a separate, already-solved data problem (JSON files supplied externally).

## 9. Open questions to confirm with Roy before building

1. Hint usage limits: unlimited per game, or capped (e.g. one letter-reveal and one definition-reveal per game)?
2. Persistence scope: one saved game per difficulty (3 concurrent saved games), or one saved game total that resets on difficulty change?
3. Should switching difficulty mid-game abandon the current game, or should difficulty only be choosable at the start of a fresh game?
